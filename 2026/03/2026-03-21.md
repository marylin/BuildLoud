## Gaps and fails

The CI was more fragile than I realized. Workflows referencing non-existent action versions, targeting branch names that do not match the default, lockfile drift after dependency bumps — none of this was caught until PRs started failing. The welcome workflow had been broken for months with nobody noticing.

PostHog integration is underwhelming — active users and events are hardcoded to zero. The integration API audit flagged this along with missing Stripe disputes, Sentry release tracking, and Railway resource metrics. Real data gaps that will matter post-launch.

Phase 3.5 (design system full application) is still pending. Some P2 test gaps remain: Playwright E2E config, route handler tests. Frontend component tests are P3.

## Lessons learned

1. Always verify GitHub Action version tags exist before referencing them — dependabot creates the PRs, but if you write a workflow by hand and guess a version number, it will silently break.
2. Unit tests should never make real HTTP calls. Use framework test utilities like Hono app.request() — the test was passing locally because the dev server happened to be running.
3. Branch naming in workflows must match the actual default branch. Renaming main to master (or vice versa) and forgetting to update workflow triggers is an easy miss.
4. CodeQL and GitHub Advanced Security features require a public repo or enterprise license. Gate these with visibility checks instead of letting them fail.
5. Lockfiles need regeneration after dependency version bumps in package.json. Frozen lockfile installs will catch this in CI, but only if CI is actually running.
6. pull_request_target with types: [opened] only fires once. Add reopened if you need to re-trigger after fixes to the base branch workflow.
